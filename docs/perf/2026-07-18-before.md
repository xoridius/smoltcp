## Repeated traffic method

The release harness was built once per feature shape:

```text
cargo build --release --example profile_loopback
cargo build --release --example profile_loopback --features socket-tcp-dynamic-buffer
```

For every shape/configuration, the exact binary command was first run once as
an unrecorded warm-up. It was then run five times in fresh processes, with each
three-second output saved as
`/tmp/smoltcp-perf-before/<shape>-<variant>-<sample>.log`. Software checksums
use no trailing flag; the offload variant appends the literal `offload`.

The complete software/offload commands, each instantiated with and without the
`offload` suffix, were:

```text
target/release/examples/profile_loopback --mode bench udp 3 [offload]
target/release/examples/profile_loopback --mode bench firehose 3 [offload]
target/release/examples/profile_loopback --mode bench pingpong 3 [offload]
```

Tables below are numeric medians of the five recorded samples. Throughput is

value; it is never reduced by lane capacity. Jain is the lower of the emitted
sent/received values in each sample, then the median. A dash means the current
harness does not emit that field for the shape.

## Downstream constrained-memory baseline

The exact required command at consumer commit `fc8193b7` was:

```text
cargo test -p netstack --release --test rss_budget_tcp --test rss_budget_combined --test rss_budget_tcp_slow --test rss_budget_tcp_rst --test rss_budget_tcp_pressure --test rss_budget_udp_pressure --test rss_budget_tcp_pool
```

It built against smoltcp `300c709a` from the lockfile and produced:

- four platform-neutral tests passed: pool budget, TCP pressure, TCP RST, and
  UDP pressure;
- three absolute jetsam/RSS tests were ignored by their source-level Linux
  guard: TCP, combined TCP/UDP, and slow TCP;
- live 1,000-connection pool budget: 24.00 MiB, versus 62.50 MiB naive and a
  50 MiB jetsam ceiling;
- pressure check: one pressure RST, then one admitted SYN-ACK after clearing
  pressure;
- cap check: four RSTs and two SYN-ACKs for six SYNs at cap two.

A cached `--nocapture` pass preserved the platform-neutral diagnostics. The
three ignored measurement bodies were also run explicitly with
`--ignored --nocapture` as supplemental Linux RSS evidence, not as Apple
jetsam truth:

| Supplemental Linux shape | Before MiB | After MiB | Delta MiB | Source budget |
|---|---:|---:|---:|---:|
| 256 TCP SYN attempts | 3.2 | 11.9 | 8.60 | 40 MiB |
| 256 TCP SYN attempts + 100 UDP | 3.4 | 13.3 | 9.93 | 50 MiB |
| 200 slow TCP SYNs, stuck upstream for 3 s | 2.6 | 25.8 | 23.21 | 50 MiB |

The supplemental tests passed on this Linux host. Actual Apple

unavailable; installed Apple Rust targets do not substitute for device

## Baseline defects that final results must improve

1. Software-checksum firehose is deterministically broken across all five
   samples: median 0.004 Gbps, 318 lane fallback allocations, 318 pool misses,
   509,768 bytes net heap delta, and 336 allocations. The five files are
   `firehose-software-1.log` through `firehose-software-5.log`. Offload
   firehose is 19.017 Gbps with zero fallback, so this is not normalized away
   as host noise.
2. Software-checksum `many_tcp_fair 3 8` has median Jain 0.9148. Every sample
   reports `uneven (no starvation)`, with individual Jain values 0.9144 to
   0.9150. The five files are `many_tcp_fair_8-software-1.log` through
   `many_tcp_fair_8-software-5.log`. Final evidence must reach Jain >= 0.95
   without hiding flows or reducing throughput.
3. The Reno and CUBIC committed netsim snapshots are stale. Their generated
   diffs are retained as raw logs only and were not accepted.
4. Multi-TCP raw RSS and allocator fields are absent, and idle/hot emits a
   misleading capacity-subtracted line alongside raw RSS. Task 10 must make
   reporting truthful and complete.
