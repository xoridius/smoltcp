The memory-reporting change was developed test-first. Its focused test first


A subsequent failure-safety regression exercised the same private
`MultiTcpWorkers` abstraction as the harness. Against its extracted barrier
implementation, an injected setup panic occurred but the bounded test failed
after 500 ms with `multi_tcp coordinator did not propagate worker failure:
Timeout`. Raw RED output is
`multi_tcp-coordinator-red-runtime.log`. The final exact setup/work tests pass:

```text
cargo test --example profile_loopback --features socket-tcp-dynamic-buffer tests::multi_tcp_coordinator_propagates_setup_panic_without_deadlock -- --exact --nocapture
cargo test --example profile_loopback --features socket-tcp-dynamic-buffer tests::multi_tcp_coordinator_propagates_work_panic_without_deadlock -- --exact --nocapture
```

The final coordinator has no fixed-count barriers. Workers report ready and
finished events; one shared predicate broadcasts start, release, or cancel to
all workers without lost wakeups. Setup/spawn/panic failures cancel and join
every started worker before the original failure is resumed or reported.
Setup, sampling, result transfer, release, and join remain outside the timed
packet loops. A panic during a timed body is therefore bounded by successful
peers reaching their finite deadline, then propagates at the next phase
boundary; no channel operation or cancellation check was added inside either
packet loop. Sockets remain alive through active-end sampling and drop only
after release.

The original ten software multi logs remain
`multi_tcp{,_sink}-preinstrument-software-{1..5}.log`, with medians 4.285 Gbps
(echo) and 9.640 Gbps (sink). The twenty barrier-coordinator samples are
preserved as `multi_tcp{,_sink}-barrier-{software,offload}-{1..5}.log`.

The final software sink median is 9.627 Gbps, -0.13% versus its original. The
unpaired echo median was 4.196 Gbps, -2.08%, so the required alternating exact
`7cedec9`/final comparison ran one warm-up each plus five pairs. Old values
`[4.263, 4.231, 4.254, 3.867, 4.139]` had a 4.231 Gbps median; final values
`[4.232, 4.299, 4.245, 4.244, 4.257]` had a 4.245 Gbps median: +0.33%, with a
+1.61% median paired change. The failure-safe harness clears the -2% software
throughput regression gate.

## Baseline defects that final results must improve

1. Software-checksum firehose is deterministically broken across all five
   samples: median 0.004 Gbps, 318 lane fallback allocations, 318 pool misses,
   509,768 bytes net heap delta, and 336 allocations. The five files are
   `firehose-software-1.log` through `firehose-software-5.log`. Offload
   firehose is 19.017 Gbps with zero fallback, so this is not normalized away
   as host noise.
2. Software-checksum `many_tcp_fair 3 8` has median Jain 0.9148. Every sample
   reports `uneven (no starvation)`, with individual Jain values 0.9144 to
   0.9150. The five files are `many_tcp_fair_8-software-1.log` through
   `many_tcp_fair_8-software-5.log`. Final evidence must reach Jain >= 0.95
   without hiding flows or reducing throughput.
3. The Reno and CUBIC committed netsim snapshots are stale. Their generated
   diffs are retained as raw logs only and were not accepted.
4. The idle/hot harness still emits a capacity-subtracted diagnostic alongside
   raw RSS. This baseline and all comparisons use only raw RSS. Multi-TCP
   reporting and failure propagation are complete without timed-loop changes.
