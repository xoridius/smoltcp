# Production validation evidence

Date: 2026-07-21

This immutable report records the matched correctness, performance, memory,
and footprint evidence for the fork's production hardening. No downstream
repository was modified or used as acceptance evidence.

## Result

All 42 ordinary-release cells and all 21 shipping-profile cells pass. The
four separate-process dynamic-buffer comparisons also pass. There is no
confirmed throughput, RSS, socket-size, allocation, fairness, or pool
regression.

The landing fixes protocol and platform defects, including:

- TCP ACK/SACK validation against transmitted sequence space, recovery through
  RecoveryPoint, overdue recovery before unsent data, per-connection congestion
  reset, exact IW10, and PAWS expiry;
- prompt dynamic receive-buffer reclamation after terminal data is drained;
- monotonic hosted `std::time::Instant` conversion;
- checked 6LoWPAN checksum, SLAAC, IPv4 fragmentation, BPF record, TUN/TAP
  descriptor, IPv6 DAD/NA, IEEE 802.15.4 MIC, and MLDv2 behavior;
- bounded, allocation-free traffic lanes and complete constrained-memory
  traffic coverage in the existing profiling harness.

## Method

The normalized comparison uses control revision
`70d84b3fc90c51a2c824b6697aad8f56e65a9991` and candidate source based on
`af4903eb1b8c0fbe285be3927b0ad660f4a8eea5`. Both variants use byte-identical
harness sources, Cargo manifest, lockfile, feature sets, and traffic commands;
only the library implementation differs. The measured source/config diff hash
is `88dc96f42774ba4a71d91add159b6b7b32e32c91e0d4a135f592d91904472799`.

The normalized matrix predates the final audit corrections. The exact landing
source, tests, snapshots, harness, and CI patch from `af4903eb1b8c0fbe285be3927b0ad660f4a8eea5`
has hash `9bd755cdcfb7679c94655e4666698c8b379513a3489d67dd40435a49732d2cc9`.
It is the SHA-256 of the binary Git diff over `Cargo.toml`, `ci.sh`, `ci/`,
`examples/`, `src/`, and `tests/snapshots/`, followed by the binary no-index
diff for the new `tests/time.rs`.
All candidate rows in the binary TSV were rebuilt from that exact patch with
Rust 1.96.0 and the same normalized control inputs.

Measurements ran on the same ten-core aarch64 Linux host with 4,096-byte pages
and Rust 1.96.0. Every primary cell has one warm-up per variant and five
alternating A/B or B/A pairs on fixed CPUs. The matrix contains 67 cells, 134
warm-ups, and 670 accepted measured processes. Twenty-nine static attempts
below 99% thread CPU were rejected, retained, and rerun.

The two noisy primary cells were repeated with the same frozen LTO binaries for
ten new alternating pairs. All 40 confirmation processes met the 99% CPU floor;
six low-CPU attempts were retained and rerun. Binary hashes were checked before
and after the confirmation.

After the final audit fixes, the exact Rust 1.96.0 landing binaries were checked
again in five alternating pairs per sensitive shape. All 30 accepted processes
met the 99% CPU floor; the exact binary hashes are in the binary TSV and the
aggregates are appended to the confirmation TSV.

The fixed gates were:

- median throughput no worse than -2%;
- RSS blocks only when the paired median exceeds both one page and 1% and the
  increase occurs in at least three of five primary pairs;
- unchanged 536-byte default and 592-byte constrained-Apple TCP sockets;
- no allocation, net-heap, or steady-pool increase for equivalent dynamic work;
- Jain fairness at least 0.95, no starvation or fallback allocation, bounded
  memory traces, and zero pool charge after teardown.

Checked aggregates are in:

- [`2026-07-18-release.tsv`](2026-07-18-release.tsv): 42 ordinary cells;
- [`2026-07-18-shipping.tsv`](2026-07-18-shipping.tsv): 21 LTO shipping cells;
- [`2026-07-18-confirmation.tsv`](2026-07-18-confirmation.tsv): the two
  ten-pair confirmations, including near-deadline `smaps`;
- [`2026-07-18-binaries.tsv`](2026-07-18-binaries.tsv): exact binary hashes and
  ELF section sizes.

The four files are covered by

### Recovery-order netsim result

The final audit found that an overdue RTO or fast retransmit could lose priority
to queued unsent data. The fix follows upstream's canonical ordering and sends
the oldest missing bytes first. Deterministic netsim output changed only in
larger-buffer loss cells; all buffer rows through 4 KiB and every zero-loss
cell are unchanged. Some high-random-loss goodput is lower because the old path
accidentally sent newer data before rewinding and retransmitting it, adding
non-protocol redundancy. The new snapshots repeated byte-for-byte and all
NoControl, CUBIC, and Reno tests pass.

Separate-process idle comparisons retain the dynamic-buffer advantage:

| Flows | Legacy RSS delta | Dynamic RSS delta | Ratio |
|---:|---:|---:|---:|
| 300 | 16,977,920 B | 196,608 B | 86.354x |
| 1,000 | 56,455,168 B | 643,072 B | 87.790x |

## Footprint

TCP `Socket` remains 536 bytes with default features and 592 bytes with the
constrained Apple feature set.

| Build | Control text+data+bss | Candidate text+data+bss | Delta |
|---|---:|---:|---:|
| release static | 545,675 B | 540,927 B | -4,748 B |
| shipping static | 458,543 B | 459,231 B | +688 B |
| release dynamic | 728,178 B | 731,702 B | +3,524 B |
| shipping dynamic | 606,471 B | 605,639 B | -832 B |
| dynamic-buffer comparator | 355,530 B | 354,310 B | -1,220 B |

These are exact-tree executable-section totals for the expanded implementation
and do not change per-socket layout. The shipping dynamic image used by the
constrained Apple profile is 832 bytes smaller. The late recovery-order fix
reduces non-LTO `Socket::dispatch` by eight bytes in both feature variants; the
larger non-LTO total shifts are linker layout/monomorphization selection. LTO,
the shipping comparison, changes by -32 bytes for static and -16 bytes for
dynamic relative to the pre-audit candidate. The comparator remains 1,220
bytes below control.

## Platform limits

Linux RSS is a repeatable supplemental signal, not Apple jetsam truth. The
repository's Apple collector reads current and lifetime-peak physical footprint
through `proc_pid_rusage(RUSAGE_INFO_V4, ...)` and is compiled for macOS, iOS
device, and iOS simulator targets. No physical iOS device or Instruments run
was available on this Linux host; on-device physical-footprint and jetsam
testing remains an integration check outside this smoltcp-only landing.
